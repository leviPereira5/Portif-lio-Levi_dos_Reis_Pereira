2. Arquitetura MVT do Django

A aplicação segue a arquitetura MVT (Model-View-Template) do Django, onde cada camada tem uma responsabilidade bem definida.

Model

Camada de dados

Responsabilidade

  • Define as tabelas da base de dados
  • Valida e persiste dados
  • Gere relações entre entidades

View

Camada de lógica

Responsabilidade

  • Recebe e processa pedidos HTTP
  • Consulta os Models
  • Passa dados para os Templates

Template

Camada de apresentação

Responsabilidade

  • Gera HTML com os dados da View
  • Usa a linguagem de templates Django
  • Herança via base.html

Fluxo de um Pedido

O que acontece quando o utilizador acede a uma página

Passos

  • Utilizador acede a um URL no browser
  • Django faz match com uma entrada em urls.py
  • A View correspondente é chamada
  • A View consulta os Models (base de dados Neon)
  • Os dados são passados ao Template
  • O HTML renderizado é devolvido ao browser

7. Arquitetura em Produção

A aplicação está alojada em ambiente universitário e usa serviços cloud para a base de dados e para os ficheiros media.

Neon

Base de dados PostgreSQL serverless

Como é usado

  • PostgreSQL na cloud, sem servidor dedicado
  • Ligação via variável DATABASE_URL no .env
  • Integrado com dj-database-url no settings.py
  • Suporta branches de base de dados para desenvolvimento

Cloudinary

Armazenamento de ficheiros media

Como é usado

  • Fotos de tecnologias, projetos, TFCs e utilizadores
  • Armazenamento persistente entre deploys
  • Integrado via django-cloudinary-storage
  • CDN automático para servir imagens rapidamente

Whitenoise

Ficheiros estáticos (CSS, JS)

Como é usado

  • Serve CSS e JS diretamente pelo Django
  • Compressão automática dos ficheiros
  • Configurado como middleware no settings.py
  • Sem necessidade de servidor externo (Nginx/Apache)

Fluxo de Dados em Produção

Como os diferentes serviços comunicam entre si

Componentes

  • Browser → Servidor WSGI (Gunicorn) → Django
  • Django ↔ Neon (leituras e escritas na base de dados)
  • Django ↔ Cloudinary (upload e leitura de ficheiros media)
  • Django → Whitenoise (ficheiros estáticos CSS/JS)
  • Credenciais isoladas em variáveis de ambiente (.env)

8. Pipeline CI/CD

Automatização do processo de integração e entrega contínua via GitHub Actions. O pipeline é ativado a cada push para o ramo main.

CI — Integração Contínua

Verificação automática do código

Passos

  • Checkout do código do repositório
  • Instalar dependências (pip install -r requirements.txt)
  • Correr testes (python manage.py test)
  • Verificar migrações não aplicadas

CD — Entrega Contínua

Deploy automático após CI passar

Passos

  • Recolher ficheiros estáticos (collectstatic)
  • Aplicar migrações (migrate)
  • Deploy no servidor universitário

Segredos e Variáveis

Credenciais protegidas no GitHub

Secrets

  • DATABASE_URL — Ligação ao Neon
  • CLOUDINARY_CLOUD_NAME
  • CLOUDINARY_API_KEY
  • CLOUDINARY_API_SECRET
  • SECRET_KEY — Chave Django
